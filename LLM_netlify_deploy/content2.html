🎯

Core Prompting Techniques

Zero-Shot

Ask directly with no examples. Relies on model's training. Best for simple, common tasks.
"Classify as positive/negative: 'I love this!'"

Few-Shot

Provide 2-5 examples to establish the pattern. Dramatically improves consistency.
"Great→Positive\nTerrible→Negative\nNice→?"

Chain-of-Thought (CoT)

Instruct the model to reason step-by-step. Essential for math and logic tasks.
"Think step by step. If John has 3 apples..."

Tree-of-Thought (ToT)

Explore multiple reasoning paths simultaneously, then pick the best. Better than single chain for complex problems.

Role Prompting

Assign a specific persona to get domain-specific responses.
"You are a senior software architect with 20 years of experience..."

Context Injection (RAG)

Provide retrieved documents as context. The foundation of all RAG systems.
"Based on this context: [docs]\nAnswer: [question]"

🎤 Interview Question

"When would you use few-shot vs zero-shot prompting?"

Answer: Zero-shot for straightforward tasks where the model's pre-training is sufficient (simple classification, summarization). Few-shot when you need specific output formatting, domain-specific terminology, or when zero-shot gives inconsistent results. Few-shot costs more tokens but significantly improves reliability for complex or nuanced tasks. Chain-of-thought is essential for any multi-step reasoning — adding "Think step by step" to math problems can improve accuracy by 30-50%.

Sampling & Generation Control

These parameters control HOW the LLM generates text. Know these to fine-tune output quality for any use case.

🌡️

Temperature — Controlling Randomness

The model predicts: "The capital of France is ___" Raw token probabilities: "Paris" → 90% "Lyon" → 5% "a" → 2% "beautiful" → 1% others → 2% Temperature = 0.0 → Always "Paris" (greedy) ✅ Use for RAG/facts Temperature = 0.7 → Usually "Paris", sometimes "Lyon" ✅ Use for chat Temperature = 1.0 → Proportional sampling ✅ Use for creative Temperature = 2.0 → Even "beautiful" has a chance ⚠️ Often incoherent
Python from groq import Groq client = Groq(api_key="your_key") prompt = "Write a one-sentence description of the moon." for temp in [0.0, 0.5, 1.0, 1.5]: r = client.chat.completions.create( model="llama-3.3-70b-versatile", messages=[{"role":"user","content":prompt}], temperature=temp, max_tokens=60) print(f"Temp {temp}: {r.choices[0].message.content}") # 0.0: "The moon is Earth's natural satellite." ← Same every time # 1.5: "The moon dances like a pearl in velvet sky." ← Creative
🎰

Top-K and Top-P (Nucleus) Sampling

Top-K Sampling

Only consider the K most likely tokens. K=1 = greedy (deterministic). K=50 = consider top 50 tokens. Simple but fixed — doesn't adapt to model confidence.

Top-P (Nucleus) Sampling

Select tokens until cumulative probability reaches P. If P=0.9: include tokens until probabilities sum to 90%. More adaptive — fewer tokens when model is confident, more when uncertain.

Context Windows & Memory

The context window is the LLM's working memory. Understanding it is critical for building real applications.

📏

Context Window — The Memory Limit

ModelContext Window≈ Words≈ Pages
GPT-3.516,385 tokens~12,000~40
GPT-4o / LLaMA 3128,000 tokens~96,000~300
Claude 3.5 Sonnet200,000 tokens~150,000~500
Gemini 1.5 Pro2,000,000 tokens~1.5M~5,000
⚠️ Critical Understanding

Context window = input + output tokens combined. If GPT-4o has 128K context and your input uses 120K, only 8K tokens remain for the response. This is exactly why RAG exists — instead of dumping entire documents into context, retrieve only 2-3 relevant chunks (~1500 tokens) and leave room for a detailed answer.

🧠

Memory Strategies in LLM Applications

LLMs are stateless — each API call is independent. To maintain conversations, you must manage memory yourself.

1. Full History (Naive)

Include all messages in every API call. Simple but hits context limit quickly. Good for short conversations (under 20 turns).

2. Sliding Window

Keep last N messages only. Simple, predictable cost. Loses early context — user may need to repeat info.

3. Summary Memory

Periodically summarize old conversation with LLM, store summary. Compress hours of chat to a few sentences. Most cost-efficient.

4. Vector Memory (RAG)

Store all conversation chunks in a vector DB. Retrieve relevant past context on each turn. Best quality, most complex.

How Tool Calling Works

User: "What's the weather in Tokyo right now?" WITHOUT tools: LLM: "I don't have access to real-time data..." ❌ WITH tools (function calling): Step 1: LLM receives tool definitions tools = [{"name": "get_weather", "description": "...", "parameters": {"city": "string"}}] Step 2: LLM generates structured tool call (NOT free text) {"tool": "get_weather", "args": {"city": "Tokyo"}} Step 3: YOUR CODE calls the actual weather API → "22°C, sunny" Step 4: Result fed back to LLM as "tool result" Step 5: LLM generates final answer: "The current weather in Tokyo is 22°C and sunny!" ✅ The LLM does NOT execute code — it generates INSTRUCTIONS for your code.
Python — Groq Tool Calling import json from groq import Groq client = Groq(api_key="your_key") tools = [{ "type": "function", "function": { "name": "get_weather", "description": "Get current weather for a city", "parameters": { "type": "object", "properties": {"city": {"type": "string"}}, "required": ["city"] } } }] response = client.chat.completions.create( model="llama-3.3-70b-versatile", messages=[{"role":"user","content":"Weather in Tokyo?"}], tools=tools, tool_choice="auto" ) # Check if LLM wants to call a tool msg = response.choices[0].message if msg.tool_calls: call = msg.tool_calls[0] args = json.loads(call.function.arguments) print(f"LLM wants to call: {call.function.name}({args})") # Your code runs the actual function here

📐

LoRA — Low-Rank Adaptation (The Smart Way)

Full fine-tuning updates all billions of parameters — very expensive. LoRA freezes the original weights and injects tiny trainable matrices.

FULL FINE-TUNING: Update W (4096 × 4096 = 16M parameters per layer) 💸 LORA: Freeze W, add two small matrices A and B W_new = W + A × B where A ∈ ℝ^(d×r), B ∈ ℝ^(r×d), rank r ≪ d Example: d=4096, rank r=16 A: 4096 × 16 = 65,536 params B: 16 × 4096 = 65,536 params Total LoRA: 131,072 params ← 99.2% fewer than full fine-tuning! Multiple LoRA adapters can be trained for different tasks and swapped on the same base model — like plugin modules.

LoRA (Standard)

Low-rank adapter matrices. Requires full-precision (FP16/BF16) base model loaded in GPU. ~2-4 GB VRAM per billion parameters. Best for mid-range GPUs (RTX 3090, A10).

QLoRA (Quantized)

Base model loaded in 4-bit precision (NF4), LoRA adapters in 16-bit. Fine-tune a 70B model on a single 24GB GPU. The technique that democratized LLM fine-tuning.

Python — QLoRA with HuggingFace from transformers import AutoModelForCausalLM, BitsAndBytesConfig from peft import LoraConfig, get_peft_model # Load model in 4-bit (QLoRA) bnb_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", # NormalFloat4 quantization bnb_4bit_compute_dtype="bfloat16" ) model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.1-8B-Instruct", quantization_config=bnb_config, device_map="auto" ) # Configure LoRA lora_config = LoraConfig( r=16, # Rank — higher = more capacity, more params lora_alpha=32, # Scaling factor target_modules=["q_proj", "k_proj", "v_proj", "o_proj"], lora_dropout=0.05, bias="none", task_type="CAUSAL_LM" ) model = get_peft_model(model, lora_config) model.print_trainable_parameters() # trainable params: 8,388,608 || all params: 8,040,000,000 || 0.10%
Chapter 11

RLHF, DPO & Model Alignment

How we teach LLMs to be helpful, harmless, and honest — the techniques behind ChatGPT and Claude's behavior.

🎯

RLHF — Reinforcement Learning from Human Feedback

STEP 1

Pre-train LLM

Train on massive text corpus — model learns to predict next tokens. Result: capable but unaligned.

STEP 2

Supervised Fine-Tuning (SFT)

Fine-tune on curated instruction-response pairs written by humans. Model learns basic instruction following.

STEP 3

Train Reward Model

Human annotators rank model outputs: "Response A is better than B." Train a neural network (reward model) to predict these human preferences.

STEP 4

PPO Fine-Tuning

Use Proximal Policy Optimization (reinforcement learning) to fine-tune LLM to maximize the reward model's score. Result: ChatGPT-like helpful behavior.

⚡

DPO — The Simpler Alternative to RLHF

Direct Preference Optimization eliminates the need for a separate reward model. It directly optimizes the LLM using preference pairs (chosen vs rejected responses).

RLHF

✅ High quality alignment
❌ Complex 4-stage pipeline
❌ Train separate reward model
❌ RL training is unstable
❌ Very compute-expensive

DPO ⭐ (Modern)

✅ Same quality as RLHF
✅ Simple: just supervised training
✅ No reward model needed
✅ Stable training
✅ 3-5× cheaper compute

🎤 Interview Question

"What is RLHF and why did DPO replace it?"

Answer: RLHF trains a reward model from human preference comparisons, then uses PPO reinforcement learning to optimize the LLM against this reward model. It works but is complex (4 stages), unstable (RL is hard), and expensive (reward model + PPO). DPO (2023) showed you can reformulate RLHF as a supervised learning problem — just train directly on (prompt, chosen, rejected) triples without RL or a separate reward model. DPO achieves comparable or better alignment with far less complexity. Most modern models (LLaMA 3, Mistral) use DPO or variants like ORPO.

An agent is not just an LLM — it's an LLM plus an environment loop:

🎤 Interview Question

"What is the ReAct pattern and why is it important?"

Answer: ReAct (Reasoning + Acting) is an agent architecture where the LLM iteratively cycles through Thought (reasoning about what to do) → Action (calling a tool) → Observation (processing the tool result). This loop repeats until the task is complete. It's important because: (1) it grounds reasoning in real-world feedback, reducing hallucinations; (2) it handles tasks requiring multiple steps and different tools; (3) the reasoning trace provides transparency and debuggability. ReAct is the foundation of how LangChain agents, OpenAI Assistants, and most production agent systems work.

Quantization — Shrinking Models to Fit on Consumer Hardware

Original LLM weights are stored as 32-bit or 16-bit floating point numbers. Quantization reduces precision to 8-bit or 4-bit integers.

Model size comparison (LLaMA 3 8B): FP32 (original): 8B × 4 bytes = 32 GB ← Needs 2× A100 GPUs FP16/BF16: 8B × 2 bytes = 16 GB ← Needs 1× A100/A10 GPU INT8 quantized: 8B × 1 byte = 8 GB ← Fits RTX 3090/4090 INT4 (GGUF Q4_K_M): 8B × 0.5B = 4 GB ← Fits in 8GB VRAM or RAM! Quality loss: FP32→INT4 causes ~1-3% quality drop on benchmarks For most practical tasks: completely acceptable trade-off

Evaluation & LLMOps

How to measure if your LLM system is actually working, and how to run it reliably in production.

📊

Evaluation Metrics

BLEU Score

Measures n-gram overlap between generated and reference text. Range 0-1. Used for translation. Limitation: doesn't capture semantic meaning.

ROUGE

Recall-Oriented Understudy for Gisting Evaluation. Used for summarization. ROUGE-L measures longest common subsequence.

Perplexity

How surprised the model is by a test corpus. Lower = better. Used to compare language models. Formula: exp(average cross-entropy loss).

LLM-as-Judge

Use GPT-4 or Claude to evaluate outputs on criteria like helpfulness, correctness, coherence. Most aligned with human preferences. Scalable alternative to human eval.

RAG-Specific Metrics (RAGAS)

MetricWhat It MeasuresRange
FaithfulnessIs the answer supported by the retrieved context? (no hallucinations)0-1 (higher better)
Answer RelevancyDoes the answer actually address the question asked?0-1 (higher better)
Context PrecisionAre the retrieved chunks relevant (not noisy)?0-1 (higher better)
Context RecallDid retrieval find all the relevant information?0-1 (higher better)
🔧

LLMOps — Production Checklist

  • Latency monitoring: Track p50, p95, p99 response times. Alert if p95 > 5s.
  • Cost tracking: Monitor token usage per request, per user, per feature.
  • Output quality evals: Regular automated evals using LLM-as-judge on sampled outputs.
  • Prompt versioning: Store prompts in version control, track changes and their impact.
  • Fallback chains: If GPT-4o fails/times out, fall back to GPT-4o-mini automatically.
  • Rate limiting: Prevent abuse, cost overruns with per-user and global limits.
  • Caching: Semantic caching — same/similar questions return cached answers (90%+ hit rate typical).
  • Guardrails: Input/output filtering for PII, toxicity, off-topic requests.
  • Streaming: Stream tokens as they generate for better UX (eliminates perceived latency).
  • Observability: Use LangSmith, Helicone, or MLflow to trace every LLM call end-to-end.

Interview Preparation — 20 Key Questions

The most commonly asked GenAI/LLM interview questions with professional-grade answers. Master these to stand out.

1. What is an LLM and how does it work at a high level?▼

An LLM is a decoder-only Transformer trained on massive text corpora to predict the next token. "Large" = billions of parameters. Training objective: given tokens T1...Tn, maximize P(Tn+1 | T1...Tn). Through scale and diverse data, emergent capabilities appear — reasoning, coding, instruction following. Inference = auto-regressive generation: one token at a time, each token fed back as input to predict the next. Temperature/top-p control sampling randomness.

2. RAG vs Fine-tuning — when to use which?▼

RAG: Best when answers need to come from specific, up-to-date documents. No training needed, citable sources, easy to update. Use for knowledge bases, doc Q&A, enterprise search. Fine-tuning: Best when changing model behavior, style, or domain vocabulary. Needs labeled data + GPU. Use for specific output formats, medical/legal jargon, consistent persona. Key: they're complementary — fine-tune for style, use RAG for knowledge.

3. What are hallucinations and how do you prevent them?▼

Hallucinations = LLM generating confident but factually incorrect output. Root cause: the model has learned to produce fluent text, not necessarily truthful text. Prevention: (1) RAG — ground responses in retrieved documents, (2) Low temperature — reduce randomness, (3) System prompt — "Only answer from provided context. Say 'I don't know' if unsure", (4) Self-consistency — generate multiple responses, take consensus, (5) Citation requirement — force model to cite sources, (6) Guardrails — post-process outputs with a fact-checker model.

4. Explain the attention mechanism in detail.▼

Self-attention allows each token to attend to all other tokens. Each token projects to Q (query), K (key), V (value) via learned matrices. Attention scores = QK^T / √d_k. Softmax normalizes to probabilities. Output = weighted sum of V. Multi-head runs H parallel attention computations, each learning different relationship types (syntax, semantics, coreference). Causal masking prevents attending to future tokens, enabling auto-regressive generation. O(n²) complexity with sequence length is the main limitation for very long contexts.

5. What is LoRA and how is it different from full fine-tuning?▼

Full fine-tuning updates all billions of parameters — requires massive GPU memory and compute. LoRA (Low-Rank Adaptation) freezes original weights and adds small low-rank matrices A (d×r) and B (r×d) to each layer. Only train A and B. For rank r=16 and d=4096: 131K params vs 16M for full layer. 99%+ reduction in trainable parameters. Benefits: Much less VRAM, faster training, can train multiple LoRA adapters for different tasks on same base model, adapters are easily swappable. QLoRA extends this by quantizing the base model to 4-bit, enabling 70B fine-tuning on a consumer GPU.

6. Explain BPE tokenization.▼

Byte Pair Encoding starts with a character-level vocabulary. Iteratively finds the most frequent adjacent pair in training corpus, merges them into a new token, and adds to vocabulary. Repeat until reaching target vocabulary size (e.g., 50K for GPT-4 uses 100,277). Result: common words are single tokens, rare words are decomposed into known subwords. "ChatGPTification" → ["Chat","G","PT","ific","ation"]. Benefits: fixed vocabulary handles any word, balances vocabulary size and sequence length, naturally handles multiple languages by sharing subword pieces.

7. What is RLHF and what problem does it solve?▼

A pre-trained LLM learns to predict text, not to be helpful or safe. RLHF (Reinforcement Learning from Human Feedback) aligns the model with human preferences. Process: (1) SFT on curated instruction-response pairs, (2) Human annotators rank model outputs, (3) Train reward model to predict human preferences, (4) Use PPO to fine-tune LLM to maximize reward score. Used in ChatGPT, Claude. DPO (2023) simplifies by directly optimizing from preference pairs without RL or a separate reward model — becoming the dominant approach today.

8. How would you build a production RAG system?▼

Ingestion Pipeline: Load docs → chunk (recursive character splitting, 512 tokens, 50 overlap) → embed (sentence-transformers or OpenAI) → store in vector DB (Pinecone/Weaviate/Qdrant) + metadata. Query Pipeline: Embed query → retrieve top-K chunks (cosine similarity) → rerank (cross-encoder) → inject into prompt → LLM generates answer. Production additions: Query expansion/HyDE, hybrid search (dense + BM25 sparse), caching semantic-similar queries, streaming responses, faithfulness eval with RAGAS, latency monitoring, fallback handling.

9. Explain Mixture of Experts (MoE) architecture.▼

MoE replaces the dense FFN layer with N expert networks + a router network. The router selects top-K experts per token (typically K=2). Only activated experts process the token — other experts are idle. Result: model has N×(dense model params) total capacity but only activates K/N of them per token. Mixtral 8×7B: 56B total, 14B active. GPT-4: estimated ~1.8T total, ~200B active. Benefits: much higher model capacity without proportional compute increase. Challenge: load balancing — need auxiliary loss to ensure all experts get trained, not just 1-2 popular ones.

10. What is the difference between encoder-only, decoder-only, and encoder-decoder Transformers?▼

Encoder-only (BERT): Bidirectional attention — each token attends to all others. Best for understanding tasks (classification, NER, embeddings). Cannot generate text. Decoder-only (GPT, LLaMA, Claude): Causal masking — each token only attends to previous. Enables auto-regressive generation. Dominant for modern LLMs due to scalability. Encoder-decoder (T5, BART): Encoder reads full input bidirectionally, decoder generates output attending to encoder states via cross-attention. Best for seq2seq: translation, summarization. Most new LLMs use decoder-only because it scales better and naturally handles in-context learning.
