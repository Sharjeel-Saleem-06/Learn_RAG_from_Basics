Chapter 01

Tokenization & Text Processing

Before an LLM can understand your sentence, it must convert it into numbers. Tokenization is the very first and most fundamental step in all of AI.

Phase 1 — Foundations

We start from absolute zero. What is a token? How does text become numbers? Why does this matter for costs and capabilities?

🔤

What Is Tokenization?

Tokenization converts raw text into a sequence of tokens, then maps them to numeric IDs. LLMs don't read letters — they read numbers. Tokenization is the bridge between human language and machine math.

"Hello, how are you?" ↓ Tokenizer ["Hello", ",", " how", " are", " you", "?"] ↓ Token IDs (GPT-4 vocabulary) [15339, 11, 1268, 527, 499, 30] ↓ Embedding Lookup [[0.12, -0.34, 0.78, ...], [0.05, 0.22, ...], ...] ← 4096-dim vectors

Key Insight: A "token" is NOT always a word. "unhappiness" → ["un", "happi", "ness"] (3 tokens). "the" → 1 token. Spaces, punctuation — each can be a token. Rule of thumb: 1 token ≈ ¾ of an English word, 4 characters.

🎤 Interview Question

"Why do LLMs use tokens instead of words or characters?"

Answer: Word-level creates millions of vocab entries, can't handle new words. Character-level has tiny vocab but loses semantic meaning and creates extremely long sequences. Subword tokenization (tokens) is the sweet spot — vocabulary of ~30K-100K, handles any word by breaking it into known subwords, balances sequence length and semantic meaning.

📐

Byte Pair Encoding (BPE) — How Tokenizers Are Built

BPE is used by GPT-4, LLaMA, Claude, Gemini. Originally a data compression algorithm adapted for NLP.

STEP 1

Start with characters

Vocabulary begins with every individual character: a, b, c, …, z, space, punctuation. ~256 entries.

STEP 2

Count most frequent pair

Count all adjacent pairs in training text. Maybe "t"+"h" appears 500,000 times — most common pair!

STEP 3

Merge pair → new token

Add "th" to vocabulary. Replace all occurrences of "t" "h" with single token "th".

STEP 4

Repeat until target vocab size

Next merge: "th"+"e"→"the", then "in", then "ing"… Stop at 50,000 tokens (GPT-4 uses 100,277).

Python import tiktoken # pip install tiktoken # Load GPT-4's tokenizer (cl100k_base) enc = tiktoken.encoding_for_model("gpt-4") text = "Machine learning is transforming healthcare" tokens = enc.encode(text) print(f"Token IDs: {tokens}") print(f"Count: {len(tokens)} tokens") print(f"Decoded pieces: {[enc.decode([t]) for t in tokens]}") # Output: # Token IDs: [13102, 6975, 374, 54952, 18985] # Count: 5 tokens # Decoded pieces: ['Machine', ' learning', ' is', ' transform', 'ing', ' healthcare'] # Token cost estimation texts = ["Hello", "Hello world", "Artificial Intelligence", "🎉🚀"] for t in texts: n = len(enc.encode(t)) cost = n * 0.000005 # GPT-4o: $5/1M input tokens print(f"'{t}' = {n} tokens = ${cost:.8f}") # Emojis cost more tokens! 🎉 = 3 tokens
💰 Tokens = Money

GPT-4o: $5 per 1M input tokens. GPT-4o-mini: $0.15 per 1M tokens. Always estimate token counts before designing your system. A typical chat message is 50-150 tokens. A 300-page book is ~150,000 tokens — that fits in GPT-4o's context window!

Chapter 02

Embeddings & Vector Representations

Embeddings are the mathematical "understanding" of language. They convert meaning into geometry — similar meanings become nearby points in space.

🧮

What Are Embeddings?

An embedding is a dense vector (a list of decimal numbers) representing the semantic meaning of text. Texts with similar meanings have vectors that are geometrically close in high-dimensional space.

Word/Sentence Embedding (simplified to 4D) ───────────────────────────────────────────────────────── "king" → [ 0.80, 0.60, -0.10, 0.90] "queen" → [ 0.75, 0.62, -0.18, 0.87] ← CLOSE to king "apple" → [-0.30, 0.10, 0.90, 0.20] ← FAR from king "banana" → [-0.22, 0.15, 0.85, 0.28] ← CLOSE to apple Famous discovery (Word2Vec, 2013): king - man + woman ≈ queen (vector arithmetic works on MEANING!) Paris - France + Italy ≈ Rome

Sparse vs Dense

Sparse (One-Hot): [0,0,0,1,0,...,0] — mostly zeros, no semantic info.
Dense (Embedding): [0.12,-0.34,0.56,...] — every dimension carries meaning. Modern LLMs always use dense.

Dimensionality

384d — all-MiniLM (free, fast)
768d — BERT-base
1536d — OpenAI ada-002
3072d — OpenAI large
More dims = more nuance but slower search

📐

Cosine Similarity — How We Measure Meaning Distance

Cosine Similarity = (A · B) / (‖A‖ × ‖B‖) Range: -1 ──────────── 0 ────────────→ +1 Opposite meaning Unrelated Identical meaning Practical Examples: "How do cars work?" vs "Mechanics of automobiles" → 0.89 ✅ Very similar "How do cars work?" vs "I love Italian pizza" → 0.08 ❌ Unrelated "happy" vs "joyful" → 0.92 ✅ Synonyms "hot" vs "cold" → 0.41 ⚠️ Related but opposite
Python from sentence_transformers import SentenceTransformer import numpy as np model = SentenceTransformer('all-MiniLM-L6-v2') # FREE, downloads automatically sentences = [ "How do cars work?", "Explain the mechanics of automobiles", "I love Italian food", ] embeddings = model.encode(sentences, normalize_embeddings=True) print(f"Embedding shape: {embeddings.shape}") # (3, 384) def cos_sim(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) for i in range(len(sentences)): for j in range(i+1, len(sentences)): sim = cos_sim(embeddings[i], embeddings[j]) print(f"{sentences[i][:20]}... vs {sentences[j][:20]}... → {sim:.4f}") # How do cars work?... vs Explain the mechani... → 0.7812 # How do cars work?... vs I love Italian food... → 0.0341
🎤 Interview Question

"Cosine similarity vs Euclidean distance — when to use which for text?"

Answer: Cosine similarity measures direction (angle between vectors) — it's scale-invariant. A short 50-word doc and a 5000-word doc about the same topic will have HIGH cosine similarity because their meaning direction is similar. Euclidean distance measures raw distance — affected by vector magnitude, so the long doc would appear different. For semantic text search = always cosine. Only use Euclidean when vector magnitudes carry meaning (e.g., word frequency counts).

⚠️ Critical Rule

The embedding model used during ingestion MUST be the same model used during query time. You cannot mix models — each model's vector space is unique. Switching models requires re-embedding your entire dataset.

Chapter 03

Attention Mechanism

The breakthrough that made modern AI possible. Attention solves the fundamental problem of "which words matter most when understanding a sentence."

🎯

The Problem Attention Solves

Before Transformers (2017), we used RNNs (Recurrent Neural Networks) which processed text word-by-word sequentially. The result: long-distance relationships were forgotten.

RNN Problem (Sequential, left-to-right): "The cat that the dog chased ran away" Processing order: The→cat→that→the→dog→chased→ran→away ↑ By "away", model ~forgot what "The cat" was! "The cat" is 8 steps from "ran" — information lost. ───────────────────────────────────────────────────────────── Attention Solution (Parallel, every-word-to-every-word): "ran" can directly look at "cat" (skip 6 words!) ✅ "chased" can directly look at "dog" ✅ "bank" can look at "river" to know it's a riverbank, not a financial bank ✅ No distance limit. No information loss. All at once.
🔑

Query, Key, Value (QKV) — The Core Mechanism

Think of attention as a smart search engine inside the model:

🔍 Query (Q)

"What am I looking for?" The current word creates a search query. Like typing "weather Tokyo" into Google.

🗝️ Key (K)

"What do I represent?" Every word advertises itself with a key. Like article titles in a catalog that Google searches through.

📦 Value (V)

"Here's my actual content." Once matched via Q×K, the value is retrieved. Like the article's actual text content.

Self-Attention Computation Step by Step: Input: "The cat sat" → Token embeddings: [e_The, e_cat, e_sat] Step 1: Create Q, K, V matrices via learned linear projections Q = embedding × W_Q (What is each word looking for?) K = embedding × W_K (What does each word contain?) V = embedding × W_V (What will each word contribute?) Step 2: Attention scores = Q × K^T Score("cat", "The") = 0.2 (low attention) Score("cat", "cat") = 0.9 (high — attends to itself) Score("cat", "sat") = 0.6 (medium — cat is what sat) Step 3: Scale scores: divide by √d_k (prevents vanishing gradients) Step 4: Softmax → probabilities summing to 1.0 Weights: [0.15, 0.55, 0.30] Step 5: Weighted sum of Values Output_cat = 0.15×V_The + 0.55×V_cat + 0.30×V_sat = Context-aware representation of "cat"
🎤 Interview Question

"Explain self-attention in one paragraph."

Answer: Self-attention allows every token to compute a weighted relationship with every other token. Each token generates three vectors: Query (searching for relevant context), Key (advertising its content), and Value (its information). Attention scores come from the dot product of each Query with all Keys, scaled by √d_k and normalized by softmax. The output for each token is a weighted sum of all Values — heavy weights on contextually relevant tokens. This enables direct long-range dependencies regardless of distance. O(n²) complexity means it's costly for very long sequences.

👁️

Multi-Head Attention — Multiple Perspectives

Instead of computing attention once, the model runs it H times in parallel, each with different learned projection matrices. Each "head" learns a different type of relationship:

Head 1 → Syntax

Learns subject-verb relationships. "The cat sat" — connects subject to verb.

Head 2 → Semantics

Learns semantic relationships. "bank" near "river" → financial bank or riverbank?

Head 3 → Coreference

Learns pronoun resolution. "John went home. He was tired." — connects "He" to "John".

Head 4 → Position

Learns local adjacency patterns. Words near each other often should be attended to together.

Outputs of all H heads are concatenated and projected back to original dimension. GPT-3 uses 96 attention heads per layer. That's 96 simultaneous "lenses" analyzing your text.

🔑 Cross-Attention vs Self-Attention

Self-Attention: A sequence attends to itself (used in decoder-only LLMs like GPT, Claude, LLaMA). Cross-Attention: Decoder attends to encoder output (used in encoder-decoder models like T5, for translation). Most modern generation LLMs are decoder-only and use only self-attention.

Chapter 04

Transformer Architecture

"Attention Is All You Need" (2017) — the paper that changed everything. Every modern LLM: GPT-4, Claude, LLaMA, Gemini is built on this architecture.

🏗️

The Decoder-Only Transformer (How GPT Works)

INPUT TEXT: "What is AI?" │ ┌──────────▼──────────┐ │ TOKENIZER │ "What is AI?" → [2061, 374, 15592, 30] └──────────┬──────────┘ │ token IDs ┌──────────▼──────────┐ │ TOKEN EMBEDDINGS │ Each ID → 4096-dim vector │ + POSITIONAL ENC │ + position info (RoPE or sinusoidal) └──────────┬──────────┘ │ ┌══════════▼══════════════════════════════╗ ║ TRANSFORMER BLOCK ×N (repeated) ║ ║ ║ ║ ┌─────────────────────────────────┐ ║ ║ │ Layer Norm (Pre-Norm) │ ║ ║ └──────────────┬──────────────────┘ ║ ║ │ ║ ║ ┌──────────────▼──────────────────┐ ║ ║ │ Multi-Head Causal Self-Attention│ ║ ← Masked: can't see future tokens ║ │ (H=32 heads, d_k=128 each) │ ║ ║ └──────────────┬──────────────────┘ ║ ║ │ + Residual connection ║ ║ ┌──────────────▼──────────────────┐ ║ ║ │ Layer Norm (Pre-Norm) │ ║ ║ └──────────────┬──────────────────┘ ║ ║ │ ║ ║ ┌──────────────▼──────────────────┐ ║ ║ │ Feed-Forward Network (FFN) │ ║ ║ │ Linear(4096→16384) + GeLU │ ║ ║ │ + Linear(16384→4096) │ ║ ║ └──────────────┬──────────────────┘ ║ ║ │ + Residual connection ║ ╚══════════════════╪═════════════════════*/ │ ┌──────────▼──────────┐ │ FINAL LAYER NORM │ └──────────┬──────────┘ ┌──────────▼──────────┐ │ LINEAR (UNEMBEDDING)│ 4096 → 100,277 vocab logits └──────────┬──────────┘ ┌──────────▼──────────┐ │ SOFTMAX │ logits → probabilities over vocabulary └──────────┬──────────┘ ┌──────────▼──────────┐ │ SAMPLE NEXT TOKEN │ temperature/top-p → pick token "Artificial" └──────────┬──────────┘ │ Append token, repeat from top! ▼ "Artificial Intelligence is..."

This repeats for every single token generated. A 200-word response ≈ 270 tokens = 270 forward passes through all N layers. That's why inference is computationally expensive and why GPUs are needed for fast generation.

📋

Every Component and Why It Exists

Component What It Does Why It's Needed
Token Embeddings Maps token ID → 4096-dim vector Converts discrete tokens into continuous space where math works
Positional Encoding Adds position info (RoPE / sinusoidal) Without it, "dog bites man" = "man bites dog" to the model
Causal Masking Prevents attending to future tokens Enables auto-regressive generation — can't "cheat" by seeing ahead
Multi-Head Attention H parallel attention computations Learns different semantic and syntactic relationships simultaneously
Residual Connections Output = f(x) + x (skip connection) Prevents vanishing gradient problem in deep networks (32+ layers)
Layer Normalization Normalizes values per token Stabilizes training, prevents exploding/vanishing activations
Feed-Forward (FFN) 2 linear + non-linearity per layer "Stores" factual knowledge; most model parameters are in FFN
Linear + Softmax Projects to vocabulary probabilities Translates internal representations to "which word comes next"
🔀

Transformer Variants — Encoder, Decoder, Both

🔵 Encoder-Only

Models: BERT, RoBERTa, DistilBERT
Task: Understanding (classification, NER, sentiment, embeddings)
How: Bidirectional attention — each token sees all others. NOT used for generation.

🟣 Decoder-Only ⭐

Models: GPT-4, Claude, LLaMA 3, Gemini
Task: Text generation (chat, code, creative writing)
How: Causal attention — each token only sees previous tokens. Industry standard today.

🟢 Encoder-Decoder

Models: T5, BART, mBART, mT5
Task: Translation, summarization (seq2seq)
How: Encoder reads all input, decoder generates output attending to encoder states.

🎤 Interview Question

"Why are modern LLMs decoder-only?"

Answer: Decoder-only models scale extremely well (empirically proven by OpenAI/Anthropic/Meta research). They naturally support in-context learning (few-shot prompting) because the model processes examples before generating — the causal attention mask means examples influence generation. The architecture is simpler (no cross-attention between encoder and decoder), and the causal masking is what enables auto-regressive token-by-token generation. Encoder-only models require separate generation mechanisms. Research also shows that very large decoder-only models can match encoder-decoder performance on most tasks.

Mixture of Experts (MoE) — How GPT-4 Scales

Traditional (Dense) models: every parameter activates for every input. MoE: only a subset of "expert" networks activate per token — getting the benefit of a massive model at the cost of a smaller one.

Standard Dense Model (7B active params for every token): Input → [ALL 7B parameters process every single token] → Output Mixture of Experts Model (e.g., Mixtral 8×7B): Input → Router Network → decides which experts to activate ↓ ┌────┬────┬────┬────┬────┬────┬────┬────┐ │E1 │E2 │E3 │E4 │E5 │E6 │E7 │E8 │ ← 8 experts └──┬─┴──┬─┴────┴────┴────┴────┴────┴────┘ ▲ ▲ Active Active (only 2 of 8 selected per token) Expert Expert Total params: 8×7B = 56B Active params per token: 2×7B = 14B ← 75% less compute! Speed: Similar to 14B dense model Quality: Similar to 56B dense model 🎉
🎤 Interview Question

"What is MoE and why is it important?"

Answer: Mixture of Experts is an architecture where a "router" network selects K of N expert sub-networks per token, instead of using all parameters. This allows dramatically larger total parameter counts with much less compute per inference. GPT-4 is believed to use MoE with ~1.8T total params but only activating ~200B per token. Mixtral 8×7B activates 14B of 56B params. Benefits: higher capacity (specialization), faster inference, lower memory bandwidth per token. Challenge: load balancing — ensuring all experts get used, not just 1-2 popular ones.
