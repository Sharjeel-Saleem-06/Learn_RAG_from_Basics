Chapter 01

What is RAG?

LLMs are brilliant, but they have a fatal flaw: they only know what they were trained on, and they make things up when they don't know.

⚠️

The Core Problem with LLMs

Imagine a brilliant professor locked in a room in 2023. They know everything up to that date, but:

  • They cannot access the internet or your private company documents.
  • They suffer from Hallucination: If you ask them a highly specific question they don't know, they will confidently invent a plausible-sounding lie.
  • Training/Fine-tuning them on new data is too slow, too expensive, and doesn't fully stop hallucinations.
💡

The Solution: Open-Book Exams

RAG (Retrieval-Augmented Generation) turns the closed-book exam into an open-book exam. Instead of asking the LLM to remember facts, we retrieve the exact facts from a database, hand them to the LLM, and say: "Read this, and answer the question based ONLY on this text."

💡 Why not just use Fine-Tuning?

A common beginner mistake is thinking: "I'll just fine-tune an LLM on my PDFs!" Fine-tuning is for teaching an LLM how to speak (style, format, domain language). It is TERRIBLE at teaching an LLM new facts. RAG is for facts. Fine-tuning is for form.

Bonus: With RAG, if a document changes, you just update the database. If you fine-tuned, you'd have to retrain the model!

Chapter 03

Data Ingestion & Chunking

Garbage in, garbage out. If your chunking strategy is bad, your RAG system will fail no matter how powerful the LLM is.

✂️

Why do we Chunk?

We cannot feed a 500-page PDF into an embedding model. Embedding models have small token limits (e.g., 512 or 8192 tokens max). Furthermore, long text dilutes the semantic meaning of the vector. A vector representing a whole book is useless for searching for a specific sentence.

Strategy How it works Pros / Cons
Fixed Character Split Splits strictly by length (e.g., every 1000 chars) with some overlap. Pro: Easy to code.
Con: Can cut sentences or words in half, destroying meaning.
Recursive Character Split Tries to split by double-newline, then newline, then space, then character. Pro: Respects paragraphs and sentences.
Con: Standard baseline, but blind to actual document structure.
Semantic Chunking Embeds individual sentences. Groups adjacent sentences that have high mathematical similarity. Pro: Keeps cohesive thoughts together. Highly accurate.
Con: Computationally expensive (requires embedding every sentence first).
Logical/Markdown Chunking Splits based on markdown headers (#, ##) or HTML tags. Pro: Respects the author's intended structure.
Con: Source docs must be well-formatted.
🔑 The Overlap Window

Always include chunk overlap (e.g., chunk size 1000, overlap 200). If a crucial sentence spans the boundary of two chunks, splitting it cleanly breaks the context. Overlap ensures context bleeds over the edges so meaning isn't lost.

Python / LangChain Style def recursive_split(text, chunk_size=1000, overlap=200): # 1. Split on \n\n (Paragraphs) # 2. If still too big, split on \n (Lines) # 3. If still too big, split on spaces (Words) # 4. Attach overlap to the end of chunk N and start of chunk N+1 pass
Chapter 04

Vector Databases (ChromaDB)

Where embeddings live. Unlike SQL databases that look for exact matches, Vector DBs search for "concepts that are mathematically close."

🗄️

How Vector Search Works (ANN)

If you have millions of documents, comparing your query vector to every single document vector (k-Nearest Neighbors / Exact Search) is too slow.

Vector databases use Approximate Nearest Neighbors (ANN). They cluster vectors using algorithms like HNSW (Hierarchical Navigable Small World), which builds a multi-layered graph. It drops onto the top layer, finds the closest cluster, drops to the next layer, and zooms in on the exact neighborhood almost instantly (logarithmic time).

ChromaDB

Local, embedded, runs in memory or SQLite. What we use in our project. Perfect for prototyping and small-to-medium datasets.

Pinecone / Weaviate / Milvus

Managed, cloud-scale vector databases. Used in production when you have billions of vectors and need high concurrency.

pgvector (PostgreSQL)

An extension for Postgres. Amazing if you already have a Postgres DB and want to combine relational queries with vector search.

🎤 Interview Question

"How does Hybrid Search work in DBs like Weaviate or Pinecone?"

Answer: Vector search (dense retrieval) is great for semantic meaning ("What is AI?"), but terrible at exact keyword matches ("Find the term 'XYZ-123'"). Hybrid Search combines Vector Search (Cosine Similarity) with traditional Keyword Search (BM25 algorithm). It runs both searches, normalizes the scores using a technique like Reciprocal Rank Fusion (RRF), and returns the combined best results. This is the industry standard for production RAG.
