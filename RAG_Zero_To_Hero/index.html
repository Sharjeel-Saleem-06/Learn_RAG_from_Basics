Chapter 01

What is RAG and Why Does it Exist?

Before understanding the solution, you must understand the problem so deeply you feel the pain yourself.

⚠️

The Fatal Flaws of LLMs (Without RAG)

Imagine GPT-4 is the world's most brilliant doctor. You've hired them to answer your patients' questions. But there's a problem:

  • Their medical knowledge is frozen at their graduation date (training cutoff). New drugs, new research? They don't know.
  • They have never read your hospital's specific policies, patient histories, or internal protocols.
  • When they don't know something, they don't say "I don't know." They confidently fabricate a plausible-sounding lie. This is called Hallucination.
⚠️ Hallucination is not a Bug — it's a Feature

LLMs are trained to generate statistically plausible next tokens. They are pattern-completion machines, not fact-retrieval machines. Hallucination is mathematically inevitable when the model is asked something outside its training distribution.

💡

The RAG Solution: From Closed-Book to Open-Book Exam

RAG (Retrieval-Augmented Generation), introduced in the 2020 Lewis et al. paper, solves this elegantly.

Instead of asking the model to remember facts from training, we retrieve the exact source documents at query time and inject them directly into the LLM's prompt:

WITHOUT RAG: User: "What is our company's refund policy?" LLM: "Your refund policy is 30 days." ← HALLUCINATED. The LLM guessed. WITH RAG: User: "What is our company's refund policy?" System: [Searches database → finds "14-day refund policy for digital goods"] Prompt: "Based on this context: '14-day refund policy for digital goods' Answer the user's question." LLM: "According to the documents, your company has a 14-day refund policy for digital goods." ← GROUNDED. Accurate. Traceable.
✅ Grounded in Facts ✅ Updatable (just update the DB) ✅ Private Data Support ✅ No Expensive Retraining
💡 RAG vs Fine-Tuning — The Classic Interview Question

Fine-Tuning teaches the model how to speak in a certain style or domain language. It's terrible at injecting new factual knowledge reliably. RAG is for injecting live, dynamic facts at inference time. They are complementary, not competing.

Chapter 02

Tokens & Embeddings — How AI Reads Text

Computers cannot understand words. Every word must be converted to numbers. This is the foundational bridge between human language and machine intelligence.

🔤

Tokenization: Breaking Text Into Pieces

An LLM does not read letter-by-letter or word-by-word. It reads tokens — sub-word units that balance vocabulary size with efficiency. The most common algorithm is Byte Pair Encoding (BPE).

Python — Tokenization Demo
import tiktoken enc = tiktoken.get_encoding("cl100k_base") # GPT-4 tokenizer text = "unhappiness" tokens = enc.encode(text) print(tokens) # → [13066, 2136] (2 tokens, not 1 word!) print(enc.decode(tokens)) # → "unhappiness" # Why this matters for RAG: # ChromaDB default model has 512 token LIMIT per chunk # GPT-4 context window is 128,000 tokens # If your chunk is 300 words ≈ 450 tokens → fits safely!
📐 Token Costs Matter

OpenAI charges per token. gpt-4o costs $0.0025 per 1K input tokens. If your retrieval returns 5 chunks of 500 tokens each, that's 2500 tokens × $0.0025 = ~$0.006 per query. At 1 million queries/day, that's $6,000/day in context costs alone. Efficient chunking saves real money.

🧠

Embeddings: Giving Text a Location in Space

An embedding is a list of floating-point numbers (a vector) that represents the meaning of a piece of text. The magic: similar meanings map to similar vectors, regardless of the exact words used.

TEXT MEANING VECTOR (simplified to 3D for illustration) ───────────────────────────────────────────────── "dog" → [0.9, 0.1, 0.2] "puppy" → [0.88, 0.12, 0.25] ← close to "dog" ✅ "cat" → [0.8, 0.6, 0.1] ← different but in "animal zone" "quantum" → [0.01, 0.9, 0.7] ← far away from all animals "K9" → [0.85, 0.15, 0.2] ← near "dog" despite different letters! Real embeddings: 768 to 3072 dimensions (OpenAI text-embedding-3-large)
Python — our embedding_generator.py
from sentence_transformers import SentenceTransformer import numpy as np model = SentenceTransformer("all-MiniLM-L6-v2") # 384-dim, free local model # Embed two semantically related sentences v1 = model.encode("The employee gets 15 vacation days per year") v2 = model.encode("How many PTO days do workers receive?") # Cosine similarity — measures angle between vectors similarity = np.dot(v1, v2) / (np.linalg.norm(v1) * np.linalg.norm(v2)) print(ff"Similarity: {similarity:.3f}") # → ~0.78 (highly similar!) # Even though the words are completely different!
Chapter 03

Chunking Strategies — The Most Underrated Decision

Chunking is the single most impactful decision in a RAG system. Bad chunks = bad answers, no matter how powerful your LLM is.

Why We Can't Feed Entire Documents

  • Embedding models have hard token limits (e.g., 512 tokens for MiniLM, 8192 for OpenAI ada-002).
  • Long texts produce "diluted" vectors. A vector for an entire book tries to represent everything, so it ends up representing nothing precisely.
  • LLM context windows cost money and have hard limits. Sending 100 pages to GPT-4 per query = 40,000+ tokens = expensive and slow.
Strategy How It Works Best For Weakness
Fixed Character Split Cut every N characters, add overlap window Quick prototypes, testing Cuts sentences mid-way, destroys meaning
Recursive Character Split Tries \n\n → \n → space → char in priority order ✅ Most text documents (baseline) Blind to document's semantic topics
Semantic Chunking Embeds every sentence; splits on cosine distance drops Research docs, unstructured text Very slow — embeds all sentences first
Markdown/Header Split Splits on # / ## / ### headers Technical docs, READMEs, wikis Needs clean, structured source
Parent-Child Split Small child chunks for search, large parent for context ✅ Production enterprise RAG More complex storage (two stores)
🔬

Our Implementation: chunker.py

chunker.py — recursive character split with overlap
from langchain.text_splitter import RecursiveCharacterTextSplitter def chunk_text(text: str) -> list[str]: splitter = RecursiveCharacterTextSplitter( chunk_size=1000, # ~700 words per chunk chunk_overlap=200, # 200-char overlap prevents context loss at boundaries separators=["\n\n", "\n", " ", ""] # priority order ) chunks = splitter.split_text(text) return chunks # Real example: # Input: 47,000 character PDF → 52 overlapping chunks # Without overlap: context lost at chunk 22/23 boundary # With overlap: answer to "…mentioned in paragraph before the table…" # bleeds into both chunk 22 AND 23 → retrieved correctly
🔑 The Golden Rule of Chunking

Your chunk size should match your expected answer size. If users ask specific, single-sentence questions → smaller chunks (300-500 chars). If users ask broad, multi-paragraph questions → larger chunks (800-1500 chars). There is no universally correct chunk size.

Chapter 04

Vector Databases — The AI's Memory Bank

A vector database doesn't look for matching keywords. It looks for matching meanings — concepts that are mathematically "close."

🗄️

The Mathematics of Similarity Search

Every chunk in your database is stored as a point in high-dimensional space. When you ask a question, the question is also converted to a vector. The database finds the N nearest neighbors using cosine similarity.

VECTOR SPACE (simplified to 2D) "cat behavior" (0.82, 0.6) ● ↗ cosine angle = 12° → very close! User Query Vector: ↗ "How do cats show affection?" ● ← Query Vector ↘ cosine angle = 85° → far away "Python syntax" (0.1, 0.95) ● Cosine Similarity = cos(θ) = (A · B) / (|A| × |B|) ∈ [-1, 1] → 1.0 = identical meaning → 0.0 = completely unrelated → -1 = opposite meanings (rare in practice)
vector_db.py — our ChromaDB implementation
import chromadb from chromadb.utils import embedding_functions client = chromadb.PersistentClient(path="./chroma_db") # auto-saves to disk embed_fn = embedding_functions.SentenceTransformerEmbeddingFunction( model_name="all-MiniLM-L6-v2" ) collection = client.get_or_create_collection( name="my_documents", embedding_function=embed_fn, metadata={"hnsw:space": "cosine"} # use cosine distance ) # Add chunks (ChromaDB auto-embeds them!) collection.add( documents=chunks, # list of text strings ids=[ff"chunk_{i}" for i in range(len(chunks))], metadatas=[{"source": "policy.pdf", "page": i} for i in range(len(chunks))] ) # Query (ChromaDB auto-embeds the query too!) results = collection.query( query_texts=["What is the vacation policy?"], n_results=3 # return top 3 most similar chunks )

Vector DB Comparison: What to Use When

🟢 ChromaDB

Embedded, local, zero setup. Stores in SQLite. What we use in this project. Perfect for prototyping and small to medium datasets (<1M chunks).

Managed cloud DB. Handles billions of vectors. Has Hybrid Search and metadata filtering built-in. Best for large-scale production.

🎤 Top Interview Question

"What are the limitations of Naive RAG?"

Answer: Naive RAG fails when: (1) The user's query is vague or poorly phrased, leading to bad retrieval. (2) The answer requires connecting information from multiple chunks (multi-hop reasoning). (3) The domain requires exact keyword matches like serial numbers or codes (vector search misses these). (4) There's no mechanism to detect if retrieved chunks are actually relevant or not. These are the problems that Advanced RAG solves.
