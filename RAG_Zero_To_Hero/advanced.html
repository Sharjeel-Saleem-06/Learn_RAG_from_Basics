Chapter 05

Query Transformation

If the user's prompt is terrible, a simple Vector Search will return terrible chunks. Advanced RAG transforms the query BEFORE searching.

🔀

Multi-Query Retrieval

Users often ask vague questions. RAG systems can suffer heavily from missing context due to a slightly misphrased question.

1. User Query: "How does the vacation policy work?" 2. Pass Query to LLM → Generate 3 variations - "What are the rules for taking time off?" - "How many PTO days do employees get?" - "Company leave and absence guidelines." 3. Embed ALL 4 queries. Run 4 vector searches. 4. Synthesize the combined unique results (remove duplicates). 5. Feed resulting chunks to LLM for the final answer.

Result: Massively increased recall. You catch documents that answer the intent but use different wording.

✨

HyDE (Hypothetical Document Embeddings)

This is arguably the most powerful query transformation technique.

1. User Query: "What is AlphaFold?" 2. Prompt LLM to write a HYPOTHETICAL answer to the query (it will likely hallucinate facts, but it will use the correct terminology and structure). Hypothetical Answer: "AlphaFold is an AI system developed by DeepMind that predicts 3D protein structures..." 3. Embed the HYPOTHETICAL ANSWER, not the user's query. 4. Search the Vector DB using that embedding.

Why it works: The user's query is a question format. The documents in the DB are answer formats (textbooks, manuals). An embedding model maps similar semantic structures closer together. By embedding a fake answer, you search the answer neighborhood of the vector space, not the question neighborhood.

👪

Parent-Child Retrieval (Auto-Merging)

A classic tradeoff: Small chunks (sentence level) have highly precise vectors for search, but they lack surrounding context. Large chunks (paragraph level) have great context for the LLM, but their vectors are muddy and hard to search for specific facts.

The Process

1. Split document into Large Parent Chunks (1000 tokens).
2. Split Parents into Small Child Chunks (100 tokens).
3. ONLY embed and index the Child Chunks.
4. When a Child is retrieved, DO NOT pass it to the LLM. Instead, look up its Parent Chunk and pass the full Parent.

Why it's amazing

You get the extremely high precision and recall of searching tiny, specific chunks, but you give the final generator the full surrounding context it needs to write a cohesive, accurate answer.

Chapter 06

Re-ranking & Filtering

Vector search is fast but coarse. Re-ranking is slow but incredibly accurate. Combining them is production-grade RAG.

⚖️

The Cross-Encoder Reranker

Standard vector search uses a Bi-Encoder format (Embed Query → Embed Doc → Compute vector distance). This is fast, but it doesn't allow the model to truly analyze the contextual interaction between the query and the document.

The Two-Stage Pipeline: STAGE 1: FAST RETRIEVAL User Query ─▶ Vector DB (Bi-Encoder) Return Top 20 coarse results (cheap, fast, good recall) STAGE 2: PRECISION RE-RANKING Feed exactly [Query + Doc N] into a Cross-Encoder Model. The Cross-Encoder reads them TOGETHER and outputs a relevance score (0.0 to 1.0). Re-score all 20 results. Take the Top 3 to give to the LLM.

Result: Cross-encoders drastically reduce hallucinations because they filter out chunks that are mathematically similar but contextually irrelevant.

💡 Standard Tech Stack

Retrieval: all-MiniLM-L6-v2 (or OpenAI Ada 002) in ChromaDB.

Re-ranking: BAAI/bge-reranker-base (or Cohere Rerank API). This single addition typically boosts RAG accuracy by 10-20%.

Chapter 07

Advanced Agentic RAG

The future of RAG isn't a straight pipeline. It's iterative, evaluative, and agentic.

🤖

Self-RAG (Self-Reflective RAG)

Standard RAG retrieves documents blindly and assumes they are good. Self-RAG uses an LLM as an evaluator at every step of the pipeline.

1. Retrieve or Not?

The LLM checks the query. "Does this need retrieval?" If the user says "Hello", it skips the DB. If they ask a fact, it retrieves.

2. Context Relevance

After retrieval, the LLM reads chunks and grades them. "Is Chunk A actually relevant to the question?" Discard irrelevant chunks.

3. Hallucination Check

After generating the final answer, a second LLM checks: "Is every fact in this generated answer supported by the retrieved chunks?"

4. Web Search Fallback

If all chunks are irrelevant (CRAG methodology), the Agent triggers a web search API instead to answer the question.

🕸️

GraphRAG (Knowledge Graphs)

Vector RAG is terrible at answering "global" questions about an entire dataset, like "What are the main themes in this 1000-page book?" Vector DBs only return isolated chunks.

GraphRAG solves this by using an LLM during the ingestion phase to extract Entities (People, Places, Concepts) and Relationships between them. It builds a Knowledge Graph (like Neo4j) and clusters communities.

Text: "Steve Jobs founded Apple with Steve Wozniak." LLM Extraction: (Steve Jobs) ───[FOUNDER_OF]──▶ (Apple) (Steve Wozniak) ─[FOUNDER_OF]──▶ (Apple) When the user queries, the system traverses the graph structure, allowing it to aggregate massive amounts of connected information instantly. You can combine Graph Search with Vector Search.
Chapter 08

Evaluation (RAGAS)

You cannot improve what you cannot measure. RAG requires an LLM-as-a-Judge pipeline.

📏

The 4 Pillars of RAG Evaluation

Using frameworks like RAGAS (Retrieval Augmented Generation Assessment) or TruLens, you pass a golden dataset (Query + Context + Expected Answer) to an LLM evaluator to score your system automatically.

Metric Where it tests The Question It Asks
Faithfulness Generation Quality Are all facts in the final answer directly inferable from the given context? (Tests for Hallucination).
Answer Relevance Generation Quality Did the generated answer actually address the user's specific question?
Context Precision Retrieval Quality Out of the chunks retrieved, were the useful ones ranked at the very top? (Tests the Vector Search/Reranker).
Context Recall Retrieval Quality Did the retrieval system find ALL the information required to answer the question? (Requires a Ground Truth answer to test against).
🎤 Interview Question

"My RAG system accurately retrieves documents, but occasionally hallucinates facts not in the documents. How do I fix this?"

Answer: First, ensure your prompt is strict (e.g., "Answer ONLY using the provided text. If unsure, say 'I don't know'"). Second, set Temperature to 0.0 to eliminate stochasticity. Third, implement a self-reflection post-processing loop where an LLM checks the output against the context (Self-RAG) before sending it to the user. Finally, check if the retrieved chunks are too small and lack the surrounding context needed to make a declarative statement.
