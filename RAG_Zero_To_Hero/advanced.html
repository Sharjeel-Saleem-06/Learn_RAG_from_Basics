Chapter 06

Query Transformation — Fix the User Before Searching

The user is the weakest link. If their question is bad, vector search returns useless chunks. Query transformation uses a lightweight LLM to improve the query before searching.

🔀

Technique 1: Multi-Query Retrieval

A single question can have many valid phrasings. Multi-Query generates 3–5 variations and searches for each.

ORIGINAL QUERY: "How does the time-off policy work?" LLM GENERATES VARIATIONS: 1. "What are the rules for taking time off at this company?" 2. "How many PTO days do employees receive per year?" 3. "What is the process to request vacation leave?" 4. "Company leave and absence guidelines" ACTIONS: → Run 4 independent vector searches → Collect top-3 results from each = up to 12 chunks → Deduplicate using exact text matching → Feed final unique set to generation LLM RESULT: Dramatically increased RECALL. You covered 4 different "directions" in vector space, catching documents that match the intent but use different words.

✨

Technique 2: HyDE — Hypothetical Document Embeddings

This is the most powerful query transformation technique. It exploits a fundamental fact of vector space geometry.

🧠 The Core Insight Behind HyDE

User queries exist in "question space" in the vector embedding model. Documents in your database exist in "answer space." These semantic neighborhoods are geometrically different. By generating a hypothetical answer first, you move from question-space into answer-space and get dramatically better search results.

STANDARD RAG: Query: "What causes quantum entanglement?" → Vector Searches: Answer documents in the DB (answer-space) Problem: Question vector is geometrically far from answer vectors! HYDE: Step 1: Send query to LLM → Generate HYPOTHETICAL answer (It may hallucinate details but uses correct terminology) Result: "Quantum entanglement occurs when two particles become correlated such that measuring one instantly affects the other, regardless of distance…" Step 2: Embed the HYPOTHETICAL ANSWER (not the user's query) Step 3: Search using this embedding WHY IT WORKS: The fake answer has same vocabulary, structure, and semantic "neighborhood" as real answer documents in DB. You search answer-space FROM answer-space! → 40-60% recall boost
🔙

Technique 3: Step-Back Prompting

Sometimes a question is too specific, and the answer lives in a broader context document. Step-Back asks the LLM to first understand the broader principle, then answer the specific question.

SPECIFIC QUERY: "Why did the iPhone XS have cellular reception issues in 2018?" STEP-BACK (Broader Question): "What factors cause poor cellular reception in smartphones?" SEARCH: Run vector search for BOTH the specific AND broad question. RESULT: The specific chunk + high-level context document are retrieved. GENERATION: LLM has both the specific answer AND the principles needed to explain WHY, producing a richer, deeper answer.
Chapter 06B

RAG Fusion — Combining Multiple Searches

Generate multiple queries, run parallel searches, then fuse the results using Reciprocal Rank Fusion. Gets the best of all retrieval paths.

How RAG Fusion Works

ORIGINAL QUERY: "What causes high blood pressure?" Step 1 — Generate Multiple Queries (LLM does this): Q1: "What causes high blood pressure?" ← original Q2: "Hypertension risk factors and causes" ← rephrased Q3: "Primary vs secondary hypertension causes" ← decomposed Q4: "Lifestyle factors that increase blood pressure" ← angle change Step 2 — Run 4 Separate Vector Searches: Search(Q1) → [Doc_A:rank1, Doc_B:rank2, Doc_C:rank3, ...] Search(Q2) → [Doc_B:rank1, Doc_A:rank3, Doc_D:rank2, ...] Search(Q3) → [Doc_C:rank1, Doc_A:rank2, Doc_E:rank3, ...] Search(Q4) → [Doc_A:rank1, Doc_D:rank2, Doc_B:rank3, ...] Step 3 — Reciprocal Rank Fusion (RRF): Score(doc) = Σ 1/(k + rank_i) where k=60 (smoothing constant) Doc_A: 1/(60+1) + 1/(60+3) + 1/(60+2) + 1/(60+1) = 0.0634 ← WINS Doc_B: 1/(60+2) + 1/(60+1) + 0 + 1/(60+3) = 0.0476 Doc_C: 1/(60+3) + 0 + 1/(60+1) + 0 = 0.0317 Step 4 — Reranked fused list → LLM generates final answer WHY IT WORKS: Doc_A appears in ALL 4 searches at high rank → truly relevant. A document that only appears in 1 search might be coincidentally ranked high.
Python — RAG Fusion with RRF 
from groq import Groq
from sentence_transformers import SentenceTransformer
import numpy as np

client = Groq(api_key="your_key")
model = SentenceTransformer('all-MiniLM-L6-v2')

def generate_queries(original_query: str, n: int = 4) -> list[str]:
    """Use LLM to generate N query variations."""
    prompt = f"""Generate {n} different versions of this question for retrieval.
Output ONLY the questions, one per line, no numbering.
Original: {original_query}"""
    
    resp = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7
    )
    queries = resp.choices[0].message.content.strip().split('
')
    return [original_query] + [q.strip() for q in queries if q.strip()]

def reciprocal_rank_fusion(ranked_lists: list[list], k: int = 60) -> list:
    """Fuse multiple ranked document lists using RRF."""
    scores = {}
    for ranked in ranked_lists:
        for rank, doc_id in enumerate(ranked):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

# Example usage
original = "What causes high blood pressure?"
queries = generate_queries(original, n=3)
print(f"Generated {len(queries)} queries:")
for q in queries:
    print(f"  - {q}")

# fused = reciprocal_rank_fusion(docs_per_query)
🎤 Interview Question

"What is RAG Fusion and when would you use it over standard RAG?"

Answer: RAG Fusion generates multiple query variations with an LLM, runs parallel vector searches for each, then combines results using Reciprocal Rank Fusion (RRF). RRF scores a document higher if it appears at high rank across multiple searches — this surfaces documents that are genuinely relevant vs coincidentally ranked high in one search. Use it when: (1) your queries are short or ambiguous, (2) retrieval precision is low in standard RAG, (3) you have enough compute budget. Standard token cost is ~3-4x higher than basic RAG since you run multiple searches and LLM calls. RRF formula: score(d) = Σ 1/(k+rank_i) where k=60 prevents very high scores for low ranks.

Query Decomposition — Break Complex Questions Down

Complex questions have multiple sub-questions. Decompose, answer each independently, then synthesize — like a researcher breaking a big question into manageable pieces.

Two Decomposition Strategies

COMPLEX QUESTION: "Compare the economic impact of COVID-19 on retail vs hospitality, and what government relief policies were most effective?" ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ STRATEGY 1: Sequential Decomposition (Answer then proceed) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Sub-Q1: "What was COVID-19's economic impact on retail?" → Search → Answer_1: "Retail saw 30% revenue drop..." Sub-Q2 (informed by A1): "What was COVID-19's economic impact on hospitality?" → Search (+ A1 as context) → Answer_2: "Hospitality suffered 60% worse..." Sub-Q3 (uses A1+A2): "What government relief policies helped retail and hospitality?" → Search (+ A1+A2 as context) → Answer_3: "PPP loans, grants, rent relief..." FINAL: Synthesize A1 + A2 + A3 → comprehensive answer ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ STRATEGY 2: Parallel Decomposition (All at once) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Sub-Q1, Sub-Q2, Sub-Q3 → Search all simultaneously → Gather all results → Synthesize in one LLM call FASTER but less contextually connected
Python — Query Decomposition 
from groq import Groq

client = Groq(api_key="your_key")

def decompose_question(question: str) -> list[str]:
    """Break a complex question into atomic sub-questions."""
    prompt = f"""You are a research assistant. Break this complex question into
simple, atomic sub-questions that can each be answered independently.
Output ONLY the sub-questions, one per line.

Question: {question}
Sub-questions:"""
    
    resp = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3
    )
    lines = resp.choices[0].message.content.strip().split('
')
    return [l.strip().lstrip('0123456789.-) ') for l in lines if l.strip()]

def answer_with_qa_pairs(question: str, prior_qa: list[tuple]) -> str:
    """Answer sub-question using previous Q&A pairs as context."""
    context = "
".join([f"Q: {q}
A: {a}" for q, a in prior_qa])
    prompt = f"""Previous findings:
{context}

Now answer: {question}"""
    
    resp = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3
    )
    return resp.choices[0].message.content

# Example: Sequential decomposition

Routing — Send Each Query to the Right Source

Not all queries belong in the same vector DB. Routing directs each query to the most appropriate data source — SQL, vector DB, web search, or direct LLM — using logic or an LLM.

Logical Routing vs Semantic Routing

USER QUERY ROUTER — Decision Tree: Incoming Query │ ┌───────────────┼───────────────┐ ▼ ▼ ▼ Logical Router Semantic Router LLM Router (rule-based) (embeddings) (prompt) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ EXAMPLE ROUTING DECISIONS: ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ "What is our return policy?" → cosine sim to "company docs" highest → Vector DB (product docs) "Show me all orders over $500 last week" → keywords: numbers, date, aggregate → SQL Database "What happened in the news about AI today?" → contains "today", "news", "latest" → Web Search Tool "What is 25% of 840?" → pure math, no retrieval needed → Direct LLM "How does RAG work?" → general knowledge question → LLM knowledge (no RAG) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ PRODUCTION: Use multiple routers in sequence: 1. from groq import Groq
from pydantic import BaseModel

client = Groq(api_key="your_key")

class RouteDecision(BaseModel):
    datasource: str  # "vectorstore" | "sql" | "web" | "llm_direct"
    reasoning: str

ROUTER_PROMPT = """Given a user question, decide which data source to use:
- vectorstore: for questions about internal documents, company policies, product info
- sql: for questions requiring aggregations, filtering, numerical data from database
- web: for questions about current events, today's news, real-time data
- llm_direct: for general knowledge, math, or reasoning that needs no external data

Question: {question}

Respond with JSON: {{"datasource": "...", "reasoning": "..."}}"""

def route_query(question: str) -> RouteDecision:
    resp = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=[{"role": "user", "content": ROUTER_PROMPT.format(question=question)}],
        temperature=0, response_format={"type": "json_object"}
    )
    import json
    data = json.loads(resp.choices[0].message.content)
    return RouteDecision(**data)

🎤 Interview Question

"How does query routing work in a production RAG system?"

Answer: Query routing classifies incoming queries and directs them to the most appropriate data source. Two main approaches: (1) Logical routing — keyword/rule-based classifiers that check for patterns like date references (→web search), SQL keywords (→database), or known topic keywords (→specific vector collection). Fast, cheap, deterministic. (2) Semantic routing — use an LLM (usually a small, fast one) to understand intent and output a structured route decision. More flexible but costs tokens. In production you chain them: first try fast logical routing, fall back to LLM routing for ambiguous queries. Routing prevents wasting expensive LLM calls on queries that can be answered with SQL or direct lookup.

Multi-Representation Indexing — Summaries for Search, Docs for Context

The search-vs-context trade-off: small chunks retrieve well but lose context; large docs have context but retrieve poorly. Solution: index small summaries for search, but return large parent docs to the LLM.

The Core Insight

PROBLEM: Small chunk (128 tokens): Retrieves precisely, but LLM sees only fragment ❌ Full document (5000 tokens): LLM has full context, but poor retrieval precision ❌ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ MULTI-REPRESENTATION INDEXING SOLUTION: ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ For each parent document: 1. Generate a dense summary (LLM-compressed, ~100-200 tokens) 2. Store summary embeddings in vector DB for retrieval 3. Store full parent document in docstore (separate storage) 4. Link summary → parent doc via ID RETRIEVAL TIME: User query → embed → find top-K summaries (vector search) │ Retrieve parent doc IDs from summaries │ Fetch full parent docs from docstore │ LLM sees FULL context while retrieval used SUMMARIES ✅ VARIANT: Proposition Indexing Instead of summaries → decompose docs into atomic propositions "The Eiffel Tower is 330 meters tall." ← one atomic fact "It was built in 1889." ← another atomic fact Each proposition is one indexable unit — maximum retrieval precision
Python — Multi-Representation Indexing 
from groq import Groq
from sentence_transformers import SentenceTransformer
import numpy as np

client = Groq(api_key="your_key")
embed_model = SentenceTransformer('all-MiniLM-L6-v2')

def generate_summary(document: str) -> str:
    """Generate a dense retrieval-optimized summary of a document."""
    resp = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=[{
            "role": "user",
            "content": f"""Summarize this document in 1-2 sentences, optimized for
search retrieval. Be specific about key entities, topics and facts.

Document: {document[:2000]}

Summary:"""
        }],
        temperature=0.2, max_tokens=200
    )
    return resp.choices[0].message.content.strip()

# Build multi-representation index
documents = [
    "Full document 1 text here... (thousands of words)",
    "Full document 2 text here... (thousands of words)",
]

# In-memory docstore (use Redis/PostgreSQL in production)
docstore = {}
summaries = []
summary_to_doc_id = {}

for i, doc in enumerate(documents):
    doc_id = f"doc_{i}"
    docstore[doc_id] = doc                    # store full doc
    summary = generate_summary(doc)           # generate summary
    summaries.append(summary)
    summary_to_doc_id[i] = doc_id
    print(f"Doc {i}: {summary[:80]}...")

# Index summaries (NOT full docs) in vector space
summary_embeddings = embed_model.encode(summaries)

def retrieve(query: str, top_k: int = 3) -> list[str]:
    """Retrieve using summaries, return full parent documents."""
    q_emb = embed_model.encode([query])
    sims = np.dot(summary_embeddings, q_emb.T).squeeze()
    top_idx = np.argsort(sims)[::-1][:top_k]
    
    # Return FULL documents, not summaries
    return [docstore[summary_to_doc_id[i]] for i in top_idx]

results = retrieve("Your search query here")
print(f"Retrieved {len(results)} full documents for LLM context")
🎤 Interview Question

"Explain Multi-Representation Indexing and Proposition Indexing."

Answer: Multi-Representation Indexing separates what you index for retrieval from what you send to the LLM for generation. You generate LLM-summarized representations of documents (100-200 tokens) and embed those for vector search, but store the full parent documents separately. At retrieval time: search using summary embeddings → get parent doc IDs → fetch full documents for the LLM context. Result: high retrieval precision (small, focused summaries) with rich generation context (full documents). Proposition Indexing goes further — decompose each document into atomic factual propositions ("The Eiffel Tower is 330m tall"), embed each proposition independently. Extreme precision at retrieval — you find exactly the right sentence — but requires more storage and pre-processing. Used in production by companies needing very high answer accuracy.

Hybrid Search — Best of Both Worlds

Vector (dense) search is great for meaning but fails at exact matches. Keyword (sparse) search is great at exact matches but misses synonyms. Combine them.

Dense vs Sparse Retrieval

Property Dense (Vector) Search Sparse (Keyword/BM25) Search
Algorithm Cosine similarity on embeddings BM25 (TF-IDF variant)
Best at Semantic meaning, synonyms, paraphrases Exact keyword matches, codes, names
Fails at Exact product codes (XYZ-99B), abbreviations Synonyms ("car" ≠ "automobile")
Example "Tell me about dog training" finds "puppy obedience" ✅ "Error code ABC-123" finds exact string ✅
HYBRID SEARCH — Reciprocal Rank Fusion (RRF) Query: "Patient medication side effects for compound XR-450" ┌──────────────────────────────────────────────────────┐ │ DENSE SEARCH (meaning) │ SPARSE SEARCH (keywords) │ │ → drug interaction docs │ → documents with "XR-450" │ │ → medication review docs │ → "compound XR" exact hit │ └──────────────────────────────────────────────────────┘ ↓ Merge via Reciprocal Rank Fusion RRF Score = Σ 1/(rank_i + k) for each retrieval system ↓ Combined results ranked by unified relevance score → Best of both: semantic context + exact term matching

Weaviate, Qdrant, Pinecone all support Hybrid Search natively. For ChromaDB, you'd implement it separately using BM25 (via rank_bm25 library) and merge the result rankings manually using RRF scoring.

Chapter 08

Re-Ranking — The Precision Layer

Vector search retrieves fast but coarsely. A Re-Ranker reads the query and document TOGETHER for surgical precision.

⚖️

Bi-Encoder vs Cross-Encoder

BI-ENCODER (Standard Vector Search — what ChromaDB does): Query → Encoder → q_vector Document → Encoder → d_vector Similarity = cosine(q_vector, d_vector) PROBLEM: Query and document are encoded INDEPENDENTLY The model never "reads" them together! Speed: O(1) at search time. ✅ Precision: Coarse. Mathematically similar ≠ contextually relevant ❌ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ CROSS-ENCODER (Re-Ranker — applied AFTER initial retrieval): Input: [QUERY] + [SEP] + [DOCUMENT] → fed TOGETHER into model Output: Relevance score (0.0 to 1.0) The model READS the interaction between query and document! Speed: Slow (runs on every chunk). Must use after pre-filtering. Precision: Extremely high. ✅ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ TWO-STAGE PIPELINE (Production Standard): Stage 1: ChromaDB retrieves top 20-50 chunks (fast, bi-encoder) Stage 2: Cross-Encoder re-scores all 20 chunks Stage 3: Take top 3-5 by cross-encoder score → feed to LLM Result: Speed of bi-encoder + precision of cross-encoder ✅
Python — Cross-Encoder Reranking
from sentence_transformers import CrossEncoder # Load cross-encoder model (runs locally, free) reranker = CrossEncoder("BAAI/bge-reranker-base") def rerank_chunks(query: str, chunks: list[str], top_k: int = 3) -> list[str]: # Pair query with every chunk pairs = [(query, chunk) for chunk in chunks] # Score all pairs (query reads each doc together) scores = reranker.predict(pairs) # Sort by score descending, return top_k ranked = sorted(zip(scores, chunks), reverse=True) return [chunk for _, chunk in ranked[:top_k]] # Usage in pipeline: rough_results = chromadb.query(question, n_results=20) # get 20 coarse final_chunks = rerank_chunks(question, rough_results, top_k=3) # filter to 3 precise

Chapter 09

Parent-Child Retrieval — The Search/Context Paradox Solved

Small chunks = precise search. Large chunks = rich LLM context. Parent-Child retrieval gives you both simultaneously.

👪

The Fundamental Tradeoff

THE DILEMMA: SMALL CHUNKS (200 tokens): ✅ Highly precise vector embeddings ✅ Great for finding the exact sentence that answers the question ❌ Too little context for LLM to write a coherent answer LARGE CHUNKS (2000 tokens): ✅ Rich context — LLM understands the full picture ❌ "Muddy" embeddings — vector contains 10 topics at once ❌ Bad retrieval precision — retrieved for wrong reasons PARENT-CHILD SOLUTION: → INDEX small child chunks (200 tokens) — for precise search → RETURN large parent chunks (2000 tokens) — for LLM context USER QUESTION → Search child chunks (precision) ✅ → Find matching child → look up its parent → Send parent chunk to LLM (context) ✅ BEST OF BOTH WORLDS! 🎯
1

Split into Parent Chunks

Take your document and create large, 1500-2000 token chunks. These are page-level passages with full context. Store them in a simple document store (dict, JSON file, or Mongo).

2

Split Parents into Child Chunks

Take each parent chunk and split it further into small 100-200 token child chunks. Each child gets a parent_id metadata field linking it back.

3

Index ONLY the Child Chunks

Embed only the small child chunks into ChromaDB. Do NOT embed the parent chunks. The vector database only contains the precise, easily-searchable small pieces.

4

Retrieve: Child Search → Parent Delivery

User asks a question → Search child chunks → Find top 3 matching children → Retrieve their parent chunks from the document store → Send parents to the LLM.

Chapter 10

Prompt Engineering for RAG

The quality of your system prompt determines whether your RAG system is helpful or dangerous. A bad prompt produces a hallucinating LLM even with perfect retrieval.

The Anatomy of a Perfect RAG Prompt

rag_pipeline.py — optimal system prompt
SYSTEM_PROMPT = """You are a precise, factual assistant. Your ONLY job is to answer the user's question using the CONTEXT provided below. STRICT RULES: 1. Answer ONLY from the Context. Do NOT use your training knowledge. 2. If the answer is not in the Context, respond: "I don't have enough information in the provided documents to answer this question." 3. Do NOT make assumptions beyond what is explicitly stated. 4. Quote the relevant part of the context to support your answer. 5. If Context contains conflicting information, mention the conflict. CONTEXT: {context} Remember: Only use the above Context. Never fabricate information.""" # Why these rules matter: # Rule 1: Prevents the LLM from "helpfully" adding training knowledge # Rule 2: Teaches graceful failure instead of hallucination # Rule 3: Stops inference chains that lead to incorrect conclusions # Rule 4: Forces the model to cite its source (auditable!)
🎤 Interview: How do you prevent RAG hallucinations?

Answer: (1) Strict system prompt with explicit "only use context" instruction. (2) Set temperature=0.0 or 0.1 to reduce creativity/randomness. (3) Implement Self-RAG post-processing — a second LLM verifies the answer against context. (4) Ensure chunk overlap to prevent context boundary artifacts. (5) Use a cross-encoder re-ranker to eliminate irrelevant chunks before they reach the LLM.

Chapter 11

Agentic RAG — When RAG Gets a Brain

Traditional RAG is a dumb pipe: Query → Search → Generate. Agentic RAG adds loops, decision-making, and self-correction to the pipeline.

The Four Agentic Patterns

🔀 Adaptive RAG (Router)

An LLM classifies the query FIRST: Is it small talk? Simple fact? Complex multi-hop? Real-time event? Routes to the appropriate handler: direct reply, vector search, web search, or a full agent chain.

🔁 Self-RAG (Self-Reflection)

After generating an answer, a second LLM check: "Is every fact in this answer supported by the retrieved context?" If not, regenerate. This is LLM-as-Judge on your own outputs.

🩺 Corrective RAG (CRAG)

After retrieval but before generation, an evaluator grades the chunks: Relevant / Ambiguous / Irrelevant. If Irrelevant → trigger web search fallback. If Ambiguous → apply query refinement. Only if Relevant → generate.

🔄 Iterative RAG

For complex multi-hop questions: Retrieve → Read → Identify what's still unknown → Retrieve again → Repeat until answer is complete. Built with LangGraph or a ReAct agent loop.

Corrective RAG (CRAG) Architecture

USER QUERY: "What is the latest breakthrough in Alzheimer's treatment in 2025?" │ ▼ Vector Search → retrieves 5 chunks from internal medical docs │ ▼ ┌─────────────────────────────────────────────────────────┐ │ EVALUATOR LLM: │ │ "Are these chunks relevant, ambiguous, or irrelevant?" │ └─────────────────────────────────────────────────────────┘ │ ├── RELEVANT: Direct to generation. Use the chunks. ✅ │ ├── AMBIGUOUS: Apply query reformulation, search again. │ └── IRRELEVANT: Internal DB lacks 2025 info (knowledge cutoff!) │ ▼ Trigger WEB SEARCH FALLBACK Serper / Tavily / Bing API → real-time 2025 data │ ▼ Summarize + filter web results │ ▼ Generate answer from web context ✅ RESULT: Your private LLM now has access to the public internet ONLY when its internal knowledge fails! Best of both. 🎯
Chapter 12

GraphRAG — Connecting the Dots

Vector RAG answers "what chunk is similar to this query?" GraphRAG answers "how are all these entities connected across the entire dataset?"

🕸️

When Vector RAG Fails — And GraphRAG Wins

QUESTION: "What are all the connections between Apple, Microsoft, and Google as revealed across these 500 internal research documents?" VECTOR RAG: Retrieves isolated chunks. Can't answer. → Returns 3 chunks about Apple's partnerships → Misses the 50 other chunks distributed across all 500 docs SCORE: ❌ Fails completely on global/aggregation questions GRAPHRAG APPROACH: Step 1: ENTITY EXTRACTION (done at ingestion time) "Apple acquired… " → (Apple) -[ACQUIRED]→ (Company X) "Tim Cook met Satya Nadella"→ (Tim Cook) -[MET]→ (Satya Nadella) "Google partnered with…" → (Google) -[PARTNER_OF]→ (Company Y) Step 2: BUILD KNOWLEDGE GRAPH (Neo4j / NetworkX) Nodes: Entities (People, Companies, Concepts, Events) Edges: Relationships (founded, acquired, competes, employs) Step 3: COMMUNITY DETECTION (Leiden algorithm) Automatically groups highly-connected nodes into clusters "Apple Community" = Apple, iPhone, Tim Cook, Cupertino, iOS, … Step 4: LLM SUMMARIZES EACH COMMUNITY → Creates hierarchical summaries from fine-detail to high-level QUERY TIME: Traverse graph → collect community summaries → LLM SCORE: ✅ Answers global aggregation questions beautifully

Comparison: When to Choose What

Query Type Vector RAG GraphRAG
"What is the policy on X?" ✅ Perfect ❌ Overkill
"Summarize all themes in 1000 docs" ❌ Fails ✅ Perfect
"How is Person A connected to Company B?" ❌ Maybe ✅ Perfect
"What product launched in January 2024?" ✅ Good ☑ Works
Setup cost Low ($$$: API calls) Very High (LLM per paragraph for extraction)
💡 Production: Combine Both!

Microsoft's GraphRAG paper (2024) showed that combining Vector RAG (for local queries) + GraphRAG (for global queries) in a hybrid router gives the best overall performance. Route specific factoid questions to vector search, and global synthesis questions to the knowledge graph.

Chapter 13

RAPTOR & ColBERT — State-of-the-Art Retrieval

Two cutting-edge techniques from academic research that dramatically improve retrieval for complex, long-context documents.

🌳

RAPTOR: Tree-Organized Retrieval

Recursive Abstractive Processing for Tree-Organized Retrieval solves the problem of needing to answer questions that require understanding multiple layers of abstraction simultaneously.

STANDARD RAG — Flat chunks, no hierarchy: [chunk 1][chunk 2][chunk 3]…[chunk 500] ← All equal, no structure Problem: "What's the overall strategic direction of Chapter 7?" → Need to read ALL of Chapter 7, not just one chunk! RAPTOR — Hierarchical Tree Structure: LEVEL 0 (Leaf): [chunk 1][chunk 2][chunk 3]…[chunk 500] (Raw chunks) ↓ K-means clustering + LLM summarization LEVEL 1: [Summary A][Summary B][Summary C]…[Summary 50] ↓ Cluster again + LLM summarize summaries LEVEL 2: [High-Level Summary 1][High-Level Summary 2]…[8] ↓ One final summary LEVEL 3 (Root): [Global Document Summary] QUERY: Collapses tree — searches ALL levels simultaneously → Specific question → finds exact leaf chunk ✅ → Broad question → finds Level 2 summary ✅ → Both at once! → Multi-level contextual answer ✅
⚡

ColBERT: Late Interaction for Deep Matching

Contextualized Late Interaction over BERT is a retrieval model that achieves cross-encoder precision at near bi-encoder speed.

BI-ENCODER (Standard): Query → [CLS] single vector Doc → [CLS] single vector Score = dot_product(q_cls, d_cls) PROBLEM: All token information is compressed into ONE vector! Much information is lost. CROSS-ENCODER (Precise but slow): [Query + Document] → single joint score READS IT TOGETHER but must re-compute for every query/doc pair! ColBERT (Best of both): Query → [q₁, q₂, q₃, q₄, q₅] ← one vector PER token Doc → [d₁, d₂, d₃, d₄, d₅] ← one vector PER token Late Interaction Score = Σᵢ max_j(qᵢ · dⱼ) → Each query token finds its best matching document token → "MaxSim" operation — fine-grained matching → Document vectors pre-computed at index time → FAST! → Token-level interaction → HIGH PRECISION!

Chapter 15

Evaluation with RAGAS — Measure Everything

You cannot improve what you cannot measure. RAG evaluation is non-trivial because answers are subjective — so we use LLMs as judges.

📏

The 4 Core RAGAS Metrics

Metric What It Measures What You Need Ideal Value
Faithfulness Are all generated facts directly supported by the retrieved context? (Hallucination detector) Generated answer + retrieved context 1.0 (100%)
Answer Relevance Did the generated answer actually address the user's question? Question + generated answer > 0.85
Context Precision Were the USEFUL retrieved chunks ranked at the TOP? (Tests Re-Ranker) Question + retrieved context + ground truth > 0.80
Context Recall Did retrieval find ALL information needed to answer? (Tests completeness) Question + retrieved context + ground truth answer > 0.80
Python — RAGAS evaluation
from ragas import evaluate from ragas.metrics import (faithfulness, answer_relevancy, context_precision, context_recall) from datasets import Dataset # Your golden test dataset data = { "question": ["What is the vacation policy?"], "answer": ["Employees get 15 PTO days per year."], # LLM generated "contexts": [["Chunk 1 text...", "Chunk 2 text..."]], # retrieved "ground_truth": ["The company provides 15 vacation days annually."] } result = evaluate( Dataset.from_dict(data), metrics=[faithfulness, answer_relevancy, context_precision, context_recall] ) print(result) # {'faithfulness': 0.95, 'answer_relevancy': 0.88, ...}
Chapter 16

Production & Deployment

Building a working demo is 20% of the job. Making it production-ready is the other 80%.

Production Checklist

💾 Caching

Cache embeddings for repeated queries using Redis or a semantic cache (GPTCache). If 30% of queries are duplicates, you eliminate 30% of API costs immediately.

🛡️ Guardrails

Input validation (detect prompt injection attempts), output validation (block harmful/PII content), and topic restriction (only answer questions within your domain).

📊 Monitoring

Track: latency per request, retrieval time vs. generation time, RAGAS metrics per day, token usage and cost per query, failure rates. Use LangSmith or Arize Phoenix.

⚡ Scaling

Move from ChromaDB to Pinecone/Weaviate for million+ vectors. Add async processing for ingestion. Use FastAPI + Uvicorn + async for concurrent query handling.

💡 Cost Optimization

Use a cheap, fast model (GPT-4o-mini, Groq Llama3) for routing and classification. Reserve expensive models (GPT-4o) only for complex generation. Token budgeting per query.

🚀 Deployment

FastAPI backend + Streamlit or React frontend. Deploy to Railway, Render, or AWS Lambda. Netlify for static frontends. Docker for reproducibility. Use GitHub Actions for CI/CD.
